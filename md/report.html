이번 세션 주요 변경사항 — literacy_level 유저 프로파일

기존에는 모든 사용자에게 동일한 깊이로 답변했음. 70대 어르신과 30대 직장인이 같은 설명을 받는 구조는 "배리어프리" 목적과 모순. literacy_level ('기초' / '일반' / '전문가') 를 세션 상태에 저장하고 에이전트 전반에 적용.

Eval 결과 분석
✅
literacy_level 변경이 신규 실패를 유발하지 않음

기존 통과 케이스들(irp_term_explanation, etf_term_explanation 등)이 모두 PASS 유지. 단일 턴 eval에서 literacy_level은 세션에 아직 없으므로 기본값 '일반'으로 동작 → 기존 동작과 동일.

⚠️
etf_price_delegation 실패 — "말씀해 주세요" 해요체 사용

실제 응답: "추가로 궁금한 점이 있으시면 말씀해 주세요." — "주세요"는 해요체로, 합쇼체 규칙(~입니다, ~합니다) 위반. instruction의 금지 목록에 ~주세요, ~세요가 누락된 사전 존재 문제. 재실행 시 통과하는 flakiness.

⚠️
macro_base_rate_query 간헐 실패 — ECOS API 불안정

helpfulness=0 실패. ECOS API 타임아웃 또는 데이터 미반환 시 "현재 해당 정보를 조회할 수 없습니다." 응답 → judge가 unhelpful 판정. API 의존성 문제이며 에이전트 로직 문제 아님.

ℹ️
단일 턴 eval의 한계 — literacy_level 효과가 eval에 반영 안 됨

현재 18개 케이스 전부 단일 턴. literacy_level은 첫 메시지에서 감지·저장 → 두 번째 메시지부터 적용. 따라서 eval만으로는 literacy_level 적응 효과를 검증할 수 없음. 멀티턴 eval 케이스 추가가 필요.

종합 평가

  • literacy_level 기반 적응형 응답 — 아키텍처 정상 동작, 신규 실패 없음
  • 실패 케이스 전부 LLM 비결정성(flakiness) 기인 — 로직 버그 아님
  • 말투 규칙(합쇼체 강제)이 flakiness의 반복적 원인 — 배리어프리 목적과 재검토 필요
  • ECOS API 의존성 — 거시지표 케이스 안정성을 위해 fallback 처리 강화 권장
  • 멀티턴 eval 케이스 추가 시 literacy_level 적응 효과 실증 가능
